  <files>src/types/benchmark.ts, src/types/benchmark.test.ts</files>

  <action>
Add schema support for truth-contract benchmark expectations.

In `src/types/benchmark.ts`, add stable enum-backed fields under `labels`, preferably a nested object such as `expectedTruth`.

Required fields:
- `actionState`: `capture_again | inconclusive | testable | ready`
- `mechanicalLevel`: `initial | intermediate | advanced | elite`
- `evidenceTier`: `weak | moderate | strong`
- `weakEvidenceDowngrade`: boolean
- `primaryFocusArea`: existing `CoachFocusArea`
- optional `secondaryFocusAreas`: array of `CoachFocusArea`
- `nextBlock`: structured expectation with `tier`, stable key/title, duration, exercise or step marker, target, minimum coverage/confidence, success/continue criteria, fail/stop criteria, and next-clip validation marker.

Keep `expectedCoachPlan` backward-compatible during the transition, but make the richer truth expectation the source for new strict checks. Do not store user-facing Portuguese action labels such as `Capturar de novo` or `Testavel` as truth. Use stable enums and let UI/copy derive labels elsewhere.

Update schema validation rules:
- `expectedTrackingTier` remains required for every clip that can count in strict/release behavior.
- if `expectedDiagnoses` is non-empty, `expectedCoachMode`, `expectedCoachPlan`, and the new truth next-block expectation are required.
- clips with `actionState=ready` must not also declare `weakEvidenceDowngrade=true`.
- clips with `weakEvidenceDowngrade=true` must expect `capture_again` or `inconclusive`, not `ready`.
  </action>

    - Schema rejects diagnosed clips that omit expected coach/truth protocol expectations.
    - Schema tests prove UI labels are not required in dataset truth.
  </acceptance_criteria>
